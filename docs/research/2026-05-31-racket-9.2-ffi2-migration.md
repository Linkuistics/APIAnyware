**Date:** 2026-05-31
**Leaf:** `update-racket-to-9.2-and-use-ffi2/020-research-racket-9.2-and-ffi2`
**Purpose:** Characterise Racket 9.2's `ffi2` library well enough that the
migration leaf (040) can be planned without re-discovering the API. Surface the
ffi2-vs-`ffi/unsafe` boundary — especially whether ffi2 covers Objective-C
message dispatch.

## 0. Source verification (leaf's mandated first step)

`https://docs.racket-lang.org/ffi2/index.html` **resolves to real content** — no
blocker. Title: *"FFI2: Racket Foreign Interface"*. Intro: *"The ffi2 library is a
more static alternative to ffi/unsafe for using libraries that have a C-based
API—without writing any new C code."* Require path `(require ffi2)`, package
`ffi2-lib`. Top-level TOC: **1 Overview · 2 Reference · 3 Interoperability with
ffi/unsafe**.

---

## 1. The load-bearing question — ObjC dispatch

**Answer: hybrid, and the hybrid is forced, not chosen.**

- The **`interop.html`** page contains **no mention** of `ffi/unsafe/objc`,
  Objective-C, `objc_msgSend`, or message dispatch. ffi2 is a C-function FFI; it
  has **no Objective-C layer**. There is no ffi2 equivalent of `tell`,
  `import-class`, `sel_registerName`, or `define-objc-class`.
- Therefore the **entire ObjC message-dispatch surface stays on
  `ffi/unsafe/objc`** — this is a finding, not a TODO. ffi2 cannot replace it.
- The repo confirms how large that surface is: `emit_class.rs` (the bulk of all
  generated bindings) is built on `import-class`, `(tell (tell X alloc) init)`,
  and typed `objc_msgSend` bindings via `get-ffi-obj`. Class/method/property
  emission is *all* ObjC dispatch.

**Recommended hybrid boundary (explicit):**

| Layer | Library | Examples in our tree |
|---|---|---|
| C-function calls (framework C exports, CoreFoundation funcs, pure C structs) | **ffi2** | `emit_functions.rs`, `emit_constants.rs` output; `CFStringCreateWithCString`; geometry structs |
| Objective-C message dispatch | **`ffi/unsafe` + `ffi/unsafe/objc`** (retained) | `emit_class.rs` output; `tell`/`objc_msgSend`; `objc-base.rkt`, `type-mapping.rkt`, `objc-interop.rkt`, `coerce.rkt` |
| IMP / callback construction for the ObjC runtime (`class_addMethod`) | **boundary** — `ffi2-callback` *can* build the pointer, but it is consumed by ObjC-runtime code | `delegate.rkt`, `objc-subclass.rkt`, `dynamic-class.rkt`, `main-thread.rkt`, `cgevent-helpers.rkt`, `block.rkt` |

**Consequence — the migration is narrower than the brief's title implies.** Most
of the generated surface and most of `runtime/` is ObjC dispatch that ffi2
cannot touch. Genuine ffi2 adoption is: the **C-function emitter**
(`emit_functions.rs` + `emit_constants.rs`), and a handful of runtime C-struct /
callback / sizeof spots. 040 should size the work against *that* surface, not
against "all FFI."

## 2. Interop / boundary mechanics (so the hybrid composes)

From `interop.html`, verbatim: *"The main incompatibility between ffi2 and
ffi/unsafe is that they have different pointer representations."* They coexist in
one program but ctypes/values from one are **not directly usable** where the
other is expected. The bridge is two conversion procedures:

- **`ptr_t->cpointer`** — ffi2 pointer → `ffi/unsafe` cpointer
- **`cpointer->ptr_t`** — `ffi/unsafe` cpointer → ffi2 pointer

`tutorial1.html`'s worked example uses exactly this to hand a Racket-`draw`
bitmap handle to a ffi2 function:
`(define bt-surface (cpointer->ptr_t (send bt get-handle)))`.

**Design implication for 040 (central tension).** Every value that crosses
between the ffi2 C-function layer and the `tell`-based ObjC layer needs an
explicit `ptr_t<->cpointer` conversion. An ObjC object (`_id`/cpointer) returned
from `tell` must become a `ptr_t` before a ffi2 function accepts it, and a
`ptr_t` returned by a ffi2 C-function must become a cpointer (and be `_id`-tagged
— see `as-id`/`coerce-arg`) before `tell` accepts it. This bridging cost is the
main thing that makes "just move functions.rkt to ffi2" non-trivial, because
many framework C-functions take/return `_id` (today `functions.rkt` already
`(require ffi/unsafe/objc)` for `_id`).

## 3. ffi2 API surface — form-by-form mapping

The migration target forms, with their ffi2 equivalents.

### 3.1 Library loading & function binding

| `ffi/unsafe` (today) | ffi2 | Notes |
|---|---|---|
| `(ffi-lib path)` | `(ffi2-lib path [version …])` | docs: *"Equivalent to ffi-lib from ffi/unsafe."* e.g. `(ffi2-lib "libcairo" '("2" #f))` |
| `(get-ffi-obj name lib type)` | `(ffi2-lib-ref name lib [#:fail …])` for the raw pointer; or bind directly with `define-ffi2-procedure` / `define-ffi2-definer` | `ffi2-lib-ref` returns a `ptr_t?` address |

**`define-ffi2-procedure`** — bind one function:
```racket
(define-ffi2-procedure id arrow-type
  #:lib lib-expr [#:c-id c-id] [#:fail fail-expr] [#:wrap wrap-expr])
```

**`define-ffi2-definer`** (recommended; the boilerplate-reducer — *this is the
right shape for our emitter*):
```racket
(define-ffi2-definer define-cairo #:lib cairo-lib)
(define-cairo cairo_create  (cairo_surface_t* . -> . cairo_t*))
(define-cairo cairo_move_to (cairo_t* double_t double_t . -> . void_t))
```
> Emitter takeaway: emit one `(define-ffi2-definer define-<fw> #:lib _fw-lib)`
> per file, then one `(define-<fw> name (args . -> . ret))` line per function —
> replacing today's per-function `(define name (get-ffi-obj … (_fun … -> …)))`.

### 3.2 Procedures / the `_fun` replacement

ffi2 uses an **arrow type** `( . -> . )` instead of `(_fun … -> …)`. Full grammar
(from `procedure.html`):
```racket
(-> arg-or-do ... maybe-varargs result maybe-errno option ...)
```
- Args: bare `int_t`, named `[arg-id : type]`, or computed `[type = auto-expr]` /
  `[arg-id : type = auto-expr]` (auto-expr may reference earlier arg-ids — this
  is ffi2's analogue of `_fun`'s `(n : _int = (length xs))` pattern).
- Result: bare type or `[result-id : result-type]`, with `#:result result-expr`
  for post-processing (see the callback example in §3.6).
- **Variadics:** `#:varargs arg-or-do ...` after the fixed args. (We emit
  variadic helpers — `runtime/variadic-helpers.rkt` — so a real equivalent
  exists; 040 to confirm exact shape.)
- `ffi2-procedure` / `ffi2-callback` are the first-class (non-`define`) forms,
  each sugar over `ffi2-cast`.

### 3.3 Base C types

ffi2 **drops the leading-underscore convention** for a **`_t` suffix**
(`base-ffi2-type.html`):

| `ffi/unsafe` | ffi2 | | `ffi/unsafe` | ffi2 |
|---|---|---|---|---|
| `_int` | `int_t` | | `_uint32` | `uint32_t` |
| `_int32` | `int32_t` | | `_uint64` | `uint64_t` |
| `_int64` | `int64_t` | | `_long` | `long_t` |
| `_double` | `double_t` | | `_bool` | `bool_t` (also `boolint_t`) |
| `_void` | `void_t` | | `_intptr` | `intptr_t` |
| `_string` | `string_t` | | `_pointer` | `ptr_t` (alias `void_t*`) |
| `_fpointer` | (a `ptr_t` subtype; callbacks return `ptr_t`) | | `_uintptr` | `uintptr_t` |

Other base types present: `int8_t uint8_t int16_t uint16_t short_t ushort_t
uint_t ulong_t size_t ssize_t byte_t float_t wchar_t intwchar_t string_utf16_t
bytes_t bytes_ptr_t path_t racket_t`, plus gcable pointer variants
`ptr_t/gcable`, `void_t*/gcable`.

> Emitter takeaway: `RacketFfiTypeMapper` (in `emit_functions.rs`/
> `shared_signatures.rs`) needs a parallel ffi2 mapping table. **Note the
> arm64 aliases** `type-mapping.rkt` defines (`_NSUInteger`→`_uint64`,
> `_NSInteger`→`_int64`) map to `uint64_t`/`int64_t`.

### 3.4 Structs (the `define-cstruct` replacement)

`define-cstruct` → **`define-ffi2-type` + `struct_t`** (`compound-ffi2-type.html`,
`tutorial2.html`):
```racket
(define-ffi2-type point_t (struct_t [x double_t] [y double_t]))
```
Generates: `point_t` (type **and** constructor), `point_t*` (pointer type),
`point_t*?` (predicate), accessors `point_t-x`/`point_t-y`, mutators
`set-point_t-x!`/…. Allocate with `(ffi2-malloc point_t)`.

Maps directly onto `type-mapping.rkt`'s `(define-cstruct _NSPoint ([x _double]
[y _double]))` family → `(define-ffi2-type NSPoint_t (struct_t [x double_t] [y
double_t]))`, with `make-NSPoint`→`NSPoint_t`, `NSPoint-x`→`NSPoint_t-x`.
**Caveat:** these structs are passed to `tell #:type`/`objc_msgSend`, so if those
calls stay on `ffi/unsafe`, the struct *type values* must remain `ffi/unsafe`
ctypes (or be bridged). 040 to decide whether geometry structs move or stay.

- **Arrays:** `(array_t elem_type count)` (count may be `*`).
- **Unions:** `(union_t maybe-tag [field type] …)`.

### 3.5 Custom type conversions (`make-ctype` / `#:scheme->c` replacement)

`define-ffi2-type` with a base/parent type + conversion options
(`define-ffi2-type.html`):
```racket
(define-ffi2-type percentage_t double_t
  #:predicate (lambda (v) (and (real? v) (<= 0.0 v 100.0)))
  #:racket->c (lambda (v) (/ v 100.0))     ; ≙ ffi/unsafe #:scheme->c
  #:c->racket (lambda (v) (* v 100.0))     ; ≙ ffi/unsafe #:c->scheme
  #:release  ...)                          ; finalize a C→Racket conversion
```
**Tagged pointer types:** giving `define-ffi2-type` a pointer parent (`ptr_t`,
`struct_t`, etc.) creates a *tagged pointer subtype* — ffi2's built-in answer to
the `_id`-tagging discipline we hand-roll in `as-id`/`coerce.rkt`
(`tutorial1.html` "Increasing Safety with Tagged Pointers"). Potentially lets the
C-function layer carry typed pointers without manual `cast`.

### 3.6 Pointers & casting

| `ffi/unsafe` | ffi2 (`pointer.html`) |
|---|---|
| `_pointer` / `cpointer?` | `ptr_t` / `ptr_t?` (alias `void_t*` / `void_t*?`) |
| `cast v from to` | `(ffi2-cast expr #:from from-type #:to to-type [#:offset …])` |
| pointer arithmetic | `(ffi2-add ptr-expr n)` (sugar over `ffi2-cast #:offset`) |
| deref / poke | `ffi2-ref` / `ffi2-set!` |
| ptr↔int | `ptr_t->uintptr_t` / `uintptr_t->ptr_t` |
| `ptr-equal?`, null checks | **not documented on `pointer.html`** → open item (§5) |

### 3.7 Callbacks (`_cprocedure` / `function-ptr` replacement)

**`ffi2-callback`** turns a Racket procedure into a C-callable function pointer
(`procedure.html`), or a nested arrow type inline (`tutorial2.html`):
```racket
(ffi2-callback proc-expr arrow-type)   ; ≙ (_cprocedure …) + function-ptr
```
Inline callback param (the cairo PNG-stream example):
```racket
(define-cairo cairo_surface_write_to_png_stream
  (cairo_surface_t* ([ptr_t ptr_t int_t . -> . int_t] ptr_t
                     . -> . (r : int_t) #:result (unless (zero? r) (error …)))))
```
**Two documented constraints that matter to us:**
1. *"A callback always runs in atomic mode … must not attempt any
   synchronization operations and generally must not raise an exception"*
   (overridable on the call side with `#:allow-callback-exn`).
2. *"A foreign callback is managed by the Racket garbage collector"* and *"must
   be retained somehow as long as it is relevant"* — same GC-pinning discipline
   our runtime already implements (module-level storage, `active-blocks` hash).

> **Open question for 040 (likely an ADR).** Our emitter and runtime carry a
> hard-won warning: `_cprocedure` callbacks **SIGILL when invoked from a non-main
> OS thread on Racket CS**, and `#:async-apply` deadlocks under
> `nsapplication-run` (see `emit_functions.rs` lines ~199–214, `cgevent-helpers`,
> `main-thread.rkt`). Does `ffi2-callback`'s atomic-mode model change this
> foreign-thread story at all? This is the single biggest unknown; it does not
> block 030 but should be spiked early in 040.

### 3.8 Memory management / finalizers

`tutorial-memory.html`: ffi2 does **not** use `register-finalizer` directly.
Instead it leans on **`ffi/unsafe/alloc`** `allocator`/`deallocator` paired via
`#:wrap`:
```racket
(define-cairo cairo_destroy (cairo_t* . -> . void_t) #:wrap (deallocator))
(define-cairo cairo_create  (cairo_surface_t* . -> . cairo_t*)
  #:wrap (allocator cairo_destroy))
```
`ffi2-malloc` modes: **`#:gcable`** (default, may relocate), **`#:manual`**
(needs `ffi2-free`), **`#:gcable-immobile`** (GC-managed but pinned — *"use when
foreign code retains pointer references across callback invocations"*).

> Runtime takeaway: `objc-base.rkt`'s `register-finalizer` + retain/release
> pattern is ObjC-lifecycle (retain/release), not C-malloc, so it **stays as-is**
> on the ObjC side. ffi2's allocator/deallocator is relevant only if a *C*
> resource in the function layer needs paired free. `#:gcable-immobile` is the
> ffi2 analogue of our manual GC-pinning for callbacks (`active-blocks`).

## 5. Per-concern takeaways → the sub-areas 040 will spawn

| Sub-area / file | Disposition | ffi2 work |
|---|---|---|
| **`emit_functions.rs`** (C-function bindings) | **migrate to ffi2** | `ffi-lib`→`ffi2-lib`; per-fn `get-ffi-obj`+`_fun` → `define-ffi2-definer`+arrow types; new ffi2 type-mapper table; bridge `_id`-typed params/results to `ptr_t` (and tag for `tell` consumers) |
| **`emit_constants.rs`** | **migrate to ffi2** (where constants are C-function-backed) | same type-mapper; keep `ffi/unsafe/objc` only if a constant is genuinely ObjC |
| **`emit_class.rs`** (class/method/property wrappers) | **stays `ffi/unsafe/objc`** | ObjC dispatch (`tell`, `import-class`, typed `objc_msgSend`) — no ffi2 equivalent. At most, the framework-dylib `ffi-lib` calls could become `ffi2-lib`, but the `objc_msgSend` `_fun` bindings stay |
| **`type-mapping.rkt`** | **hybrid** | geometry `define-cstruct`→`struct_t` *if* their consumers move; NSString/NSArray/NSDictionary conversions use `tell`/`objc_msgSend` → stay `ffi/unsafe/objc` |
| **`coerce.rkt`, `objc-base.rkt`, `objc-interop.rkt`** | **stays `ffi/unsafe/objc`** | `cast _pointer _id`, `tell`, retain/release, `register-finalizer` — all ObjC lifecycle/dispatch. `objc-interop.rkt` is the curated re-export allowlist; update its comments to name the retained boundary |
| **`block.rkt`** | **candidate** | `_cprocedure`/`function-ptr`→`ffi2-callback`; `define-cstruct`→`struct_t`; **`ctype-sizeof` needs a ffi2 equivalent** (not found in fetched pages — §open) |
| **`delegate.rkt`, `objc-subclass.rkt`, `dynamic-class.rkt`** | **boundary, defer** | IMP construction (`_cprocedure`+`function-ptr` for `class_addMethod`); `ffi2-callback` could build the pointer but consumers are ObjC-runtime; gated on the atomic-mode/foreign-thread question (§3.7) |
| **`main-thread.rkt`, `cgevent-helpers.rkt`** | **boundary, defer** | `_cprocedure` callbacks invoked from run-loop/main-thread; same thread-safety unknown |

### Open items (carry into 040; some may become ADRs)

1. **Callback thread-safety under ffi2** — does `ffi2-callback`'s atomic-mode
   change the Racket-CS foreign-thread SIGILL / `#:async-apply`-deadlock
   situation? *Biggest unknown.* Spike early. → likely an **ADR** (changes
   approach).
2. **`ctype-sizeof` ffi2 equivalent** — needed by `block.rkt`. Not on the fetched
   pages; check `compound-ffi2-type`/`pointer` reference more deeply in 040.
3. **`ptr-equal?` / null-check equivalents** — not documented on `pointer.html`;
   our runtime uses `ptr-equal?` heavily. Confirm ffi2 form or keep via interop.
4. **Geometry-struct disposition** — move `_NSPoint`-family to `struct_t`, or
   keep `ffi/unsafe` so they compose with `tell #:type`? Driven by item below.
5. **Maximize ffi2 adoption + native performance (directive, 2026-05-31 — not a
   sizing question).** The value proposition is settled: on the C-function side
   of the seam, adopt ffi2's richest idioms everywhere they fit (tagged pointer
   subtypes, custom `#:racket->c`/`#:c->racket` conversions, `struct_t`,
   `define-ffi2-definer`, allocator/deallocator), and push heavy work into the
   native `libAPIAnywareRacket.dylib` so the Racket-side ffi2 bridge stays thin
   and static (max performance — a fat native core behind a thin static seam).
   The ObjC surface dominating is a *boundary* fact, **not** a reason to minimize
   ffi2 on the C side. 040 plans toward maximum ffi2 + native, not minimum churn.
   Concrete candidates: move geometry structs to `struct_t` (item 4 → yes);
   evaluate batching `string<->NSString` / `list<->NSArray` conversions
   (`type-mapping.rkt`, today many per-element `tell` calls) into native helpers.
