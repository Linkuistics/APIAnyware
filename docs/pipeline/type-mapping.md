**2026-05-21 — distilled from the archived Ravel-Lite `memory.yaml` (observations made over sessions through 2026-04-28; each verified against the current tree):**

- 🔴 `get_enum_constant_value()` returns `Option<(i64, u64)>`. For a signed underlying type use `.0` (i64); for an unsigned underlying type (`CF_ENUM(uint32_t, …)`) clang sign-extends top-bit-set values into the i64 component, so `.0` is wrong (e.g. `0xFFFFFFFE` → `-2`). `extract_declarations.rs` calls `get_canonical_type()` on the underlying type to determine signedness, takes `.1` (u64) for unsigned types, and skips constants exceeding `i64::MAX` with a warning. Canary: `kCGEventTapDisabledByTimeout` should emit `4294967294`, not `-2`.
- 🔴 `Entity::get_enum_underlying_type()` on a `CF_ENUM(UInt32, …)` returns a `Type` whose `TypeKind` is `Typedef` (the `UInt32` alias), not `UInt`. Calling `map_primitive_name` without `get_canonical_type()` falls through to the display-name default → `"UInt32"`, which the FFI mapper does not recognise. Always call `.get_canonical_type()` to drill through typedef wrappers to the real `TypeKind`; `enum_underlying_primitive()` in `type_mapping.rs` does this.
- 🔴 `TypeRefKind::Alias { name, framework, underlying_primitive: Option<String> }` stores the canonical primitive name (e.g. `"uint32"`, `"int64"`) for enum-typedef aliases. The FFI mapper consults `underlying_primitive` to pick the correct FFI width and signedness — without it, enum-typedef aliases silently map to a wrong-width FFI type. The shared helper `racket_ffi_type_for_primitive` converts the canonical name to an FFI type string and is called from both the `Alias` and `Primitive` arms; the contract mapper consults the same field so signed enum typedefs emit `exact-integer?` rather than `exact-nonnegative-integer?`.
- 🔴 `TypeAlias` (a type declaration) and `TypeNameAlias` (a type reference in a signature) are distinct Swift API digester node kinds. C typedefs like `CFStringEncoding` appear as `TypeNameAlias` nodes whose first child is a `TypeNominal` with the resolved type; `map_swift_type()` in `extract-swift/src/type_mapping.rs` recurses into that first child. Without recursion, framework-qualified primitive names (e.g. `CoreFoundation.CFStringEncoding`) reach downstream unrecognised → `_pointer`. A childless `TypeNameAlias` falls through to `Primitive`.
- 🟡 `map_typedef` in `extract-objc/src/type_mapping.rs` must check whether a `TypeKind::Pointer` pointee is `FunctionPrototype` and extract full param/return signatures into `TypeRefKind::FunctionPointer`. Without this, callback typedefs like `CGEventTapCallBack` resolve to a plain `Pointer`. Any new `TypeRefKind` variant requires updating the exhaustive matches in `ffi_type_mapping.rs` and every emitter crate.
- 🟡 `TypeRefKind::CString` is a dedicated variant for `const char *` parameters and returns only; non-const `char *` (output buffers) stays `Pointer`. FFI type `_string`; return contract `(or/c string? #f)`; parameter contract `string?`. Any new language emitter must handle `CString` in its FFI and contract mappers.
- 🟢 `is_generic_type_param` in `generation/crates/emit/src/ffi_type_mapping.rs` identifies generic type parameters by the heuristic "starts with an uppercase letter followed by a lowercase letter" (e.g. `ObjectType`, `KeyType`). No false positives in the current framework set; theoretically vulnerable to a single-uppercase-letter framework prefix, of which none exist today.
- 🟡 `Primitive` matching in any language FFI mapper cannot use exact lowercase: `swift-api-digester` produces framework-qualified PascalCase (`Swift.Void`, `Swift.Bool`, `Swift.Double`) while ObjC uses uppercase `BOOL`. `normalize_primitive_name` strips the framework prefix via `rsplit_once('.')` and lowercases. Every new language target's FFI mapper must apply the same normalization, in `map_type()` and in trait methods like `is_void()`.
- 🟡 Three code paths govern geometry-struct handling in `ffi_type_mapping.rs`: `is_known_geometry_struct()` (recognition), `map_geometry_struct_alias()` (Alias kind → language type), and the `Struct` match arm (delegates to `map_geometry_struct_alias()`). The current recognised set is NSRect/CGRect, NSPoint/CGPoint, NSSize/CGSize, NSRange, NSEdgeInsets, NSDirectionalEdgeInsets, NSAffineTransformStruct, CGAffineTransform, CGVector. Adding a new geometry struct requires updating all three sites plus the corresponding `define-cstruct` in the Racket runtime `type-mapping.rkt`.
