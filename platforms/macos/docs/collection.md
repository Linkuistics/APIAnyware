# Collection Pipeline Learnings

**2026-03-31:**
- 🔴 Typedef aliases (e.g., `NSImageName`) must be resolved to canonical types at collection time in `type_mapping.rs`. ObjC object pointer typedefs → `Id`/`Class`, primitive typedefs → `Primitive`, enum/struct typedefs → keep as `Alias`. Without this, the FFI mapper defaults to `_uint64` for unrecognized aliases, which crashes at runtime for object-type parameters.
- 🟡 Category property merging must deduplicate by name — `class.properties.extend(category_props)` creates duplicates when a property exists on both the class interface and a category. Fixed with HashSet filter in `extract_declarations.rs`.

**2026-05-21 — distilled from the archived Ravel-Lite `memory.yaml` (observations made over sessions through 2026-04-28; each verified against the current tree):**

- 🔴 `clang-2.0.0` panics (process abort) on any non-UTF-8 `CXString`, not just file paths. Doc comments are the highest-risk source — some Quartz-subframework cursors carry a stray non-UTF-8 byte in their brief comment. The panic occurs inside `extern "C" fn visit`, so the Rust runtime converts it to an abort that bypasses `catch_unwind` — unless `catch_unwind(AssertUnwindSafe(…))` wraps the call before it crosses the FFI boundary. Pattern: `safe_get_comment_brief` / `catch_clang_utf8_panic` in `extract_declarations.rs`. Reapply for any new hot spot that calls a libclang string-returning method (name, USR, display_name, comment variants) on entities from the Quartz umbrella or other headers with miscoded bytes.
- 🔴 `clang_tokenize` returns a NULL token buffer for `UnexposedAttr` cursors backed by macro expansions (e.g., `NS_SWIFT_UI_ACTOR`). The NULL dereference aborts the process across the `extern "C"` visitor boundary — `catch_unwind` cannot intercept it. ObjC attribute macros must not be read via `clang_tokenize`; use source-line scanning instead (see next entry).
- 🟡 The `ObjCInterfaceDecl` cursor extent starts at `@interface`; attribute macro lines above it (e.g., `NS_SWIFT_UI_ACTOR`) are outside the extent and unreachable via any libclang cursor API. `detect_class_swift_attributes` reads the source file directly, scanning upward from the `@interface` line for `*_SWIFT_UI_ACTOR` / `*_SWIFT_MAIN_ACTOR` token patterns. It matches by suffix (`ends_with("_SWIFT_UI_ACTOR")`) so future framework-prefixed aliases are picked up without code changes. Reuse this approach for any future pre-interface annotation capture.
- 🔴 `CF_ENUM`/`NS_ENUM` macros expand to a forward-declaration cursor followed by the definition cursor, both with the same name. A `HashSet`-based dedup guard on enum names must check `entity.is_definition()` before inserting — inserting on the first (forward-decl) visit causes the definition to be skipped silently, yielding zero-value enum sections. The `EnumDecl` and `StructDecl` arms in `extract_declarations.rs` both gate on `entity.is_definition()`. `ObjCInterfaceDecl` does not need the guard — `@class Foo;` forward declarations produce `ObjCClassRef` cursors, not `ObjCInterfaceDecl`.
- 🔴 `API_UNAVAILABLE(macos)` marks declarations absent from the macOS dylib even though they have `Linkage::External` and ordinary USRs — neither the linkage filter nor any USR filter catches them. `is_unavailable_on_macos(entity)` iterates `entity.get_platform_availability()` for `platform == "macos" || "macosx"` with `unavailable: true`; missing availability data → `false`. The filter applies at four granularities: `extract_class`, `extract_protocol`, `extract_method`, `extract_property`. Class/protocol hits are wholesale drops. Verify unexplained runtime "symbol not found" errors against `dyld_info -exports <dylib>`.
- 🟡 `extract_constant` / `extract_function` skip declarations with `entity.get_linkage() == Some(Linkage::Internal)`. `static const` variables and `static inline` functions are inlined at use sites and emit no dylib symbol — extracting them causes runtime `get-ffi-obj` failures. Canonical examples: `NSHashTableStrongMemory`, `NSHashTableWeakMemory`.
- 🟡 libclang surfaces `#define` macros as `EntityKind::MacroDefinition`, never reaching extract-objc's `VarDecl`/`FunctionDecl` arms — except CFSTR macros, extracted separately via `extract_cfstr_macro_constant`. Non-CFSTR macros in ObjC headers reach collected IR only via `swift-api-digester`, re-exported as top-level `Var` nodes with `c:@macro@` USRs. The filter for macro-derived constants lives in `extract-swift/src/declaration_mapping.rs` (the `PREPROCESSOR_MACRO` skip reason), not in extract-objc.
- 🟡 `apianyware_types::skipped_symbol_reason` holds the `&'static str` tags used by both extractors (`internal_linkage`, `platform_unavailable_macos`, `swift_native`, `preprocessor_macro`, `anonymous_enum_member`). Format: `"<tag>: <human description>"`; audit tooling matches via `reason.contains(tag)`. When adding a new filter, extend this module first — do not inline the reason string at the filter site. A symbol can appear in both the IR (kept by one extractor) and `skipped_symbols` (filtered by the other) — `skipped_symbols` membership does not mean "absent from final IR".
- 🔴 Three USR prefix families in `swift-api-digester` output produce no dylib symbol and must be filtered by `non_c_linkable_skip_reason()` in `extract-swift/src/declaration_mapping.rs`: `s:` (Swift-native, callable only via the Swift ABI); `c:@macro@` (preprocessor macro cursor); and `c:@Ea@…` / `c:@EA@…` (anonymous C enum members — value inlined, no dylib export). The single-character case difference (`Ea` vs `EA`) means both prefixes must be filtered together — filtering only one leaks ~95% of the symbols. Named-enum members (`c:@E@<Enum>@<Member>`) are routed through `Enum` → `EnumElement`, not this filter.
- 🔴 `swift-api-digester` re-emits extension targets as foreign top-level type decls whose `moduleName` points at the owning module, not the framework under extraction. Without a filter, `map_abi_to_framework` attributes those extension members to the wrong framework, and the resolve layer's `effective_method` rule then propagates the spurious methods to every conforming class across all frameworks. `is_foreign_module_type_decl()` drops top-level `Class`/`Protocol`/`Struct`/`Enum` nodes whose `moduleName` differs from the framework root — but keeps them for cross-import overlay frameworks (`_<A>_<B>`), where the foreign type decls *are* the overlay's API surface. `Func`/`Var` nodes always carry the correct owning module and are unaffected.
- 🟡 A triggering symptom underpredicts a filter's scope unpredictably (one canary → 1,738 removed symbols across 56 frameworks). Before committing an extraction filter: stash the change, full re-collect (`cargo run -p apianyware-collect`, ~2 min) for pre-filter counts, restore, re-collect, diff per-framework. Measure semantic removals, not structural proxies like `dyld_info` counts.
- 🟢 `IGNORED_FRAMEWORKS` in `collection/crates/extract-objc/src/sdk.rs` excludes `DriverKit` (C++ kernel headers, libclang parsing fails in a user-space context) and `Tk` (legacy Tcl/Tk X11 toolkit, not a native macOS framework). Extend this list for other frameworks that genuinely fail libclang extraction rather than papering over the errors.
- 🟢 libclang synthesizes names for anonymous enums from the first member, embedding the absolute SDK path. These appear in IR `enums[].name` and produce cosmetic diffs when IR is generated under different Xcode installations or SDK locations. Not a correctness issue — values and types are unaffected.
- 🟡 CFSTR-backed constants (e.g., `kCFBundleNameKey`) are extracted via the `MacroDefinition` arm, which calls `extract_cfstr_macro_constant` — it tokenizes the macro's source range and matches `CFSTR ( "literal" )`, setting `macro_value`. Fragile if Apple changes the CFSTR macro shape. Non-CFSTR macros hit this arm and return `None`; a `seen` set prevents double-counting when `swift-api-digester` also emits the same constant.
- 🟢 `EnumDecl` child cursors feed `enums[].values`, not `constants[]`. No extract-objc code path lets anonymous enum members reach IR `constants[]`; the corresponding non-C-linkable filter in extract-swift (`c:@Ea@` / `c:@EA@`) handles the same symbols from the digester side.
- 🟡 Headers outside the standard `.framework` tree (e.g., libdispatch in `/usr/include/dispatch/`) use a four-part synthetic-framework pattern: (1) a checked-in umbrella header at `extract-objc/synthetic-frameworks/<name>/<name>.h`; (2) `sdk.rs::synthetic_frameworks()` appends a synthetic `FrameworkInfo`; (3) `is_from_framework()` branches on the synthetic name to accept the relevant `usr/include/<subdir>/` paths; (4) the emitter's `framework_ffi_lib_arg` maps the synthetic name to the real dylib short name. Use this pattern for any future system-header extraction lacking a `.framework` directory.
