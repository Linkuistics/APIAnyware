# Analysis Pipeline Learnings

**2026-05-21 — distilled from the archived Ravel-Lite `memory.yaml` (observations made over sessions through 2026-04-28; each verified against the current tree):**

- 🔴 A fresh `collected/` can coexist with a stale `resolved/` (or `annotated/`, `enriched/`) if a downstream stage was not re-run after an extraction fix landed. Symptom: a canary absent from `collected/*.json` but still present in `resolved/*.json`. Before forming any hypothesis from the IR, compare the `analysis/ir/` mtimes to `collection/crates/extract-{swift,objc}/src/**` mtimes — if source is newer, regenerate downstream first. `analysis/scripts/regenerate-stale-pipeline.sh` automates the check. `collection/ir/` is gitignored, so re-run the collector before grepping JSON to confirm a fix.
- 🟡 The freshness rule for each pipeline stage is newest-input-mtime > newest-output-mtime (not oldest-output). The oldest-output variant produces perpetual false positives from tool-managed files (e.g. `raco make` `.zo`/`.dep`) or orphaned prior-version outputs; exclude tool-managed subdirectories (e.g. `*/compiled/*`) from the output scan. A stage with no existing output directory is treated as fresh, not stale — stub emitters register in the emitter registry but may never have produced a `generated/` directory, and treating absence as staleness creates an infinite regen loop.
- 🟡 LLM annotations use two dedicated paths: `analysis/ir/llm-summaries/` (gitignored; intermediate `.methods.json` files from `llm-extract`, derivable from `resolved/` IR) and `analysis/ir/llm-annotations/*.llm.json` (**versioned in git** — <1 MB total for 284 frameworks; regenerating from scratch costs ~14M tokens). Workflow: `llm-extract` → `.methods.json` → per-framework Claude Code subagents (parallel) → each runs `llm-validate` pre-return → `.llm.json` → `annotate --llm-dir analysis/ir/llm-annotations/` → `enrich`. A fresh clone needs no LLM calls — just run `annotate --llm-dir …`.
- 🟡 `validate_llm_annotations` / `validate_llm_file` in `analysis/crates/annotate/src/llm.rs` perform semantic cross-referencing beyond serde's structural checks. `ValidationReport` carries `errors` and `warnings`; `is_ok()` is errors-only — warnings never fail validation. `llm-validate` exits 0 on valid input (including warnings), 1 with errors. Subagents must run `llm-validate` before returning a `.llm.json`, and must omit classes with zero annotated methods from the `classes` array entirely (empty class objects cause validation failures).
- 🔴 `merge_annotations` in `analysis/crates/annotate/src/validate.rs` uses per-param union for `parameter_ownership`: heuristic entries for `param_index` values absent from the LLM list are appended; LLM wins on conflict. A whole-list clone (the intuitive approach) silently drops heuristic entries for params the LLM did not annotate — WebKit canary: 67 ownership annotations with per-param union vs 15 with a whole-list clobber. Scalar fields (`threading`, `error_pattern`) use empty-fallback (LLM value if non-empty, else heuristic). `block_parameters` currently uses all-or-nothing list-replace safely; apply per-element union if divergence is ever observed.
- 🟡 `annotate` without `--llm-dir` falls back to the prior annotated checkpoint via `load_existing_llm_annotations`, applying `retain_llm_sourced` — a pure filter that drops `source = Heuristic` entries and prunes classes with no remaining methods, returning `None` when nothing LLM-sourced survives. A heuristic-only rerun therefore produces only `source = Heuristic` output with no LLM-source contamination.
- 🔴 `.llm.json` files become stale when an extraction fix changes the method set for affected frameworks — `llm-validate` then reports methods referenced in the file but absent from current IR. Detection: `analysis/scripts/check-llm-annotation-drift.sh` (pass `--skip-regen` to use existing summaries) runs `llm-validate` on every `.llm.json`. Run it after any extraction change affecting method counts. Unlike stale pipeline checkpoints — which a re-run fixes — LLM annotation staleness requires new LLM subagent dispatches to refresh the affected files.
- 🟡 Protocol-inherited methods are propagated to class IR at the resolve layer via `transitively_conforms_to` (reflexive-transitive closure over `conforms_to` / `protocol_inherits`) and an `effective_method` Datalog rule in `analysis/crates/resolve/src/program.rs`. Because the fix lives at the resolve layer — not per-emitter — every current and future language emitter inherits protocol-method propagation automatically. Cross-framework indexes (`build_method_index`, `build_property_index`) span all loaded frameworks; a single-framework-scoped index loses metadata for inherited methods declared in another framework.
- 🟡 `annotate_method_heuristic` in `analysis/crates/annotate/src/heuristics.rs` takes `&Class`, not `class_name: &str`. Class-level signals — `swift_attributes` (MainActor detection) and `properties` (copy-block-setter detection) — are reachable only via the full `Class` struct. Future heuristics needing class-level data must keep this signature.
- 🟡 `ir::Class.swift_attributes: Vec<String>` is populated from two sources — `swift-api-digester` `declAttributes` (via `extract-swift`'s `map_class`) and source-line scanning of ObjC headers (via `detect_class_swift_attributes`) — merged with set-union semantics in `merge_class_members`. `derive_threading` in `heuristics.rs` calls `is_main_actor_attribute(attr)`, which strips the `_Concurrency.` module prefix, so it detects both bare `"MainActor"` and `"_Concurrency.MainActor"`.
- 🟡 `ir::Property.is_copy: bool` (from libclang's copy attribute, serialised only when true) enables a heuristic that classifies `@property (copy)` block-setter calls as `stored` lifecycle. `is_copy_block_property_setter` maps a synthesised setter selector (`set<Cap><Rest>:`) back to the property name and checks `is_copy && Block type`. Without it, copy-block setters are misclassified `async_copied`, under-pinning GC retention.
- 🟢 Block-invocation reclassification is the primary LLM value-add: heuristics emit zero `stored` block-lifecycle annotations for most frameworks; the LLM identifies stored handlers (AppKit 0→25, CoreData 0→11, AVFoundation +15). `stored` blocks need GC-prevention pinning that `async_copied` blocks do not — misclassification causes intermittent observer disappearance under GC pressure. Threading yield from the LLM declines as heuristics mature (class-level `_SWIFT_UI_ACTOR` propagation); ownership yield is near-zero (heuristics cover the standard `setDelegate:` / `addObserver:` patterns).
- 🟡 `is_observer_param` in `heuristics.rs` classifies the first parameter of `add…Observer:` selectors as `weak` ownership. Structural rule: the first selector segment `starts_with("add") && ends_with("Observer")`. It is generative — `addObserver:`, `addSharedObserver:forKey:options:context:`, future `addXxxObserver:` variants all match — and correctly excludes block-form `addObserverForName:object:queue:usingBlock:` (first segment `addObserverForName` does not end with `Observer`).
- 🟡 Code-generated frameworks (Matter, and likely the CryptoTokenKit `TKToken*` cluster, LDAP, GSS) have mechanically structured selectors that permit template-based bulk annotation instead of per-method dispatch: identify selector-family regex patterns and annotate by pattern group. Budget `35k + ~5k × num_unique_patterns`, not `35k + N × 250` — Matter calibrated at 12,827 methods / 7 patterns / ~57k tokens (~50× under the naive projection). Strategy documented in `analysis/scripts/llm-annotate-orchestration.md`.
- 🟢 LLM annotation subagents are parallel-dispatch-safe: 25 simultaneous subagents returned `llm-validate warnings=0` with zero report divergences, ~105s wall-clock vs ~1,100s serial-aggregate (~10× speedup). Use the dispatch loop in `analysis/scripts/llm-annotate-orchestration.md`.
- 🟢 To identify LLM-redundant frameworks, run `annotate` without `--llm-dir` (heuristic-only) and find where the output equals merged output exactly — those need no LLM call. Counting `source == "llm"` entries in merged output is the wrong proxy (LLM precedence wins even on equal values). `audit-llm-redundancy.py` implements the correct comparison.
- 🟡 `developer.apple.com` returns only `<title>` to an automated fetch (JS-rendered SPA). The MacOSX SDK headers at `MacOSX.sdk/System/Library/Frameworks/<F>.framework/Headers/` are the authoritative source for LLM annotation — they carry `weak`/`copy`/`assign`/`nullable` attributes and doc comments unavailable from the web. Annotation subagent prompts must prioritise SDK headers over web docs.
- 🟡 `build_enrichment_data()` / `build_verification_report()` filter by `framework_classes`, computed in `build_enriched_framework()`; without filtering, violations from all frameworks bleed into every report. Delegate-protocol attribution filters by `framework.protocols` (declaration site), not `framework_classes` (usage site). Rule: filter by where an entity is *declared*, not where it is *used*.
